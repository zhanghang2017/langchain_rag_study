# Backend API Spec (MVP)

Base URL: `http://localhost:3001`

Version Prefix: `/v1`

Auth: none (MVP stage)

Content-Type: `application/json` (except upload endpoint)

Success:

```json
{
  "data": {}
}
```

Error:

```json
{
  "error": {

    "message": "Invalid request body",
    "details": [
      {
        "path": "userId",
        "message": "Required"
      }
    ]
  }
}
```

## User

### POST /v1/users/identify (Optional Bootstrap)

Description: optional bootstrap endpoint. It can be used to pre-create/get user by client user ID. Current implementation uses browser fingerprint as `userId`. If frontend already sends `userId` in business endpoints (upload/create session), this call can be skipped.

Request Body:

```json
{
  "userId": "fp_xxx"
}
```

Compatibility:

- Backend still accepts legacy field `browserFingerprintHash`, but `userId` is the canonical field name.

Response `200`:

```json
{
  "data": {
    "id": "fp_xxx"
  }
}
```

## File + Ingestion Task

### POST /v1/files/upload

Description: upload binary file via multipart, then server verifies optional SHA-256 hash, computes MD5, deduplicates by `(userId, contentMd5)`, and creates ingestion task asynchronously when needed.

Request Content-Type:

- `multipart/form-data`

Form fields:

- `userId` (required, string; current value is browser fingerprint)
- `contentHash` (optional, 64-char SHA-256 hex string, recommended for direct upload integrity verification)
- `file` (required, binary)

Notes:

- `storagePath` is generated by server, not provided by frontend.
- Current implementation writes upload to disk temp path first, then computes SHA-256/MD5 by stream to support large files.
- Final persisted path is organized by user directory: `upload/<userId>/<contentMd5>.<ext>`.

Example (conceptual):

```json
{
  "userId": "fp_xxx",
  "contentHash": "8d969eef6ecad3c29a3a629280e686cff8fab9f5dbeebd7f1b1e6f6df2a6c5f7",
  "file": "<binary>"
}
```

Response `201`:

```json
{
  "data": {
    "deduplicated": false,
    "file": {
      "id": "uuid",
      "userId": "fp_xxx",
      "storagePath": "upload/fp_xxx/0f343b0931126a20f133d67c2b018a3b.pdf"
    },
    "task": {
      "id": "uuid"
    }
  }
}
```

Response `201` (deduplicated):

```json
{
  "data": {
    "deduplicated": true,
    "file": {
      "id": "uuid",
      "userId": "fp_xxx",
      "storagePath": "upload/fp_xxx/0f343b0931126a20f133d67c2b018a3b.pdf"
    },
    "task": null
  }
}
```

### Chunked Upload APIs (Large File / Resume)

#### POST /v1/files/upload/chunked/init

Description: initialize one resumable chunk upload session.

Request Body:

```json
{
  "userId": "fp_xxx",
  "uploadId": "8d969eef6ecad3c29a3a629280e686cff8fab9f5dbeebd7f1b1e6f6df2a6c5f7",
  "fileName": "manual.pdf",
  "fileSizeBytes": 104857600,
  "totalChunks": 40
}
```

Response `201`:

```json
{
  "data": {
    "uploadId": "8d969eef6ecad3c29a3a629280e686cff8fab9f5dbeebd7f1b1e6f6df2a6c5f7",
    "totalChunks": 40,
    "uploadedChunks": []
  }
}
```

#### POST /v1/files/upload/chunked/chunk

Description: upload one chunk file.

Request Content-Type:

- `multipart/form-data`

Form fields:

- `uploadId` (required, 64-char SHA-256 hex string)
- `chunkIndex` (required, integer, starts at 0)
- `chunk` (required, binary)

Response `200`:

```json
{
  "data": {
    "uploadId": "8d969eef6ecad3c29a3a629280e686cff8fab9f5dbeebd7f1b1e6f6df2a6c5f7",
    "totalChunks": 40,
    "uploadedChunks": [0, 1, 2]
  }
}
```

#### GET /v1/files/upload/chunked/status

Description: query uploaded chunk indexes for resume.

Query Params:

- `uploadId` (required, 64-char SHA-256 hex string)

Response `200`:

```json
{
  "data": {
    "uploadId": "8d969eef6ecad3c29a3a629280e686cff8fab9f5dbeebd7f1b1e6f6df2a6c5f7",
    "totalChunks": 40,
    "uploadedChunks": [0, 1, 2]
  }
}
```

#### POST /v1/files/upload/chunked/complete

Description: merge all uploaded chunks, then run server-side md5 dedupe + ingestion task creation.

Request Body:

```json
{
  "uploadId": "8d969eef6ecad3c29a3a629280e686cff8fab9f5dbeebd7f1b1e6f6df2a6c5f7"
}
```

Response `201`: same as `/v1/files/upload` response shape.

### Frontend Upload Calling Protocol

This section defines the recommended client-side flow for large-file upload, resume, and retry.

1. Decide upload strategy

- If file size is small (for example <= 20 MB), client can call POST /v1/files/upload directly.
- Client should compute SHA-256 before direct upload and send it as `contentHash` for server-side integrity verification.
- If file size is large, client should use the chunked upload APIs below.

2. Initialize upload session

- Call POST /v1/files/upload/chunked/init with:
  - userId
  - uploadId (file SHA-256)
  - fileName
  - fileSizeBytes
  - totalChunks
- Persist uploadId locally (memory + localStorage/IndexedDB) for resume.

3. Upload chunks (can be concurrent)

- Split file into fixed chunks.
- Recommended chunk size: 2 MB to 8 MB.
- Recommended parallelism: 3 to 5 concurrent requests.
- For each chunk call POST /v1/files/upload/chunked/chunk with:
  - uploadId
  - chunkIndex
  - chunk (binary)

4. Retry policy

- On network/server transient failure, retry current chunk.
- Recommended retry count: up to 3 times per chunk.
- Recommended backoff: 500 ms, 1000 ms, 2000 ms.
- If a chunk still fails after max retries, pause upload and keep uploadId for resume.

5. Resume policy (breakpoint continuation)

- On app refresh/reopen, call GET /v1/files/upload/chunked/status?uploadId=...
- Read uploadedChunks and skip uploaded indices.
- Continue uploading only missing chunk indices.

6. Complete upload

- After all indices [0..totalChunks-1] are uploaded, call POST /v1/files/upload/chunked/complete.
- Server merges chunks, computes MD5, deduplicates, writes metadata, and creates ingestion task as needed.

7. Post-complete handling

- If response contains deduplicated=true:
  - task is null
  - file is already known by content, upload is effectively completed.
- If deduplicated=false:
  - poll GET /v1/tasks/:taskId to track ingestion progress.

8. UI state recommendations

- Show two progress bars if possible:
  - Upload progress: uploaded bytes / total file bytes.
  - Ingestion progress: task progress from GET /v1/tasks/:taskId.
- Keep uploadId bound to a file fingerprint (name + size + lastModified) on client for resume UX.
- Server stores final files under `upload/<userId>/...`.

9. Suggested completion criteria

- Upload phase completed when complete API returns 201.
- Knowledge readiness completed when:
  - task status is success, and
  - related file parseStatus is success.

### GET /v1/files

Description: list files for one fingerprint user.

Query Params:

- `userId` (required, string)
- `parseStatus` (optional, enum: `queued|running|success|failed|cancelled`)
- `limit` (optional, int, default `50`, range `1-200`)

Compatibility:

- Query param `browserFingerprintHash` is still accepted for backward compatibility.

Response `200`:

```json
{
  "data": [
    {
      "id": "uuid",
      "userId": "fp_xxx",
      "fileName": "manual.pdf",
      "fileSizeBytes": 2048,
      "storagePath": "upload/fp_xxx/0f343b0931126a20f133d67c2b018a3b.pdf",
      "parseStatus": "success",
      "uploadedAt": "2026-03-31T10:00:00.000Z"
    }
  ]
}
```

### GET /v1/tasks/:taskId

Description: get ingestion task status.

Path Params:

- `taskId` (required, UUID)

Response `200`:

```json
{
  "data": {
    "id": "uuid",
    "userId": "fp_xxx",
    "fileId": "uuid",
    "status": "running",
    "progress": 35,
    "errorMessage": null,
    "createdAt": "2026-03-31T10:00:00.000Z"
  }
}
```

Error `404`:

- `TASK_NOT_FOUND`

## AI Callback

### POST /v1/ai/ingestion/callback

Description: worker callback endpoint for ingestion status updates.

Request Body:

```json
{
  "taskId": "uuid",
  "status": "running",
  "progress": 55,
  "errorMessage": "optional"
}
```

Field rules:

- `taskId` required, UUID
- `status` required, enum: `queued|running|success|failed|cancelled`
- `progress` optional, number in `[0, 100]`
- `errorMessage` optional, string

Response `200`:

```json
{
  "data": {
    "task": {},
    "file": {}
  }
}
```

## Error Code Table

- `VALIDATION_ERROR`: request body/query/path does not satisfy schema.
- `TASK_NOT_FOUND`: task id not found.
- `SESSION_NOT_FOUND`: session id not found.
- `RESOURCE_NOT_FOUND`: generic DB not found fallback.
- `INTERNAL_SERVER_ERROR`: unhandled error.
- `UPLOAD_STORAGE_ERROR`: upload middleware did not produce file path.
- `CONTENT_HASH_MISMATCH`: direct-upload file hash does not match server-computed SHA-256.
- `UPLOAD_SESSION_NOT_FOUND`: chunk upload session not found.
- `CHUNK_UPLOAD_INCOMPLETE`: some chunk indexes are missing when completing upload.

## Ingestion State Transition

After `POST /v1/files/upload`, backend returns success immediately for new files, then relies on AI callback updates:

1. initial task `queued`, file `queued`
2. callback `running` -> file `running`
3. callback `success` -> file `success`

Failure path:

- callback `failed` or `cancelled` -> file uses the same terminal status
